# Results Consistency Protocol — EMBA Thesis
**Version:** 1.0  
**Scope:** All quantitative results, tables, and interpretive text in `thesis_final.md` and `results/thesis_results.md`  
**Last updated:** 2026-05-19

---

## HOW TO USE THIS DOCUMENT

This protocol governs the verification of numerical and interpretive consistency across all result-bearing files in the repository. It is executed after every pipeline run (`8_statistical_analysis.py`) that regenerates `results/thesis_results.md`, and before any commit that modifies `thesis_final.md` chapter text.

The protocol comprises four sequential passes:

1. **Pass A – Source-of-truth audit**: confirm which files are authoritative for each result type.
2. **Pass B – Cross-file numerical consistency**: verify that numbers in `thesis_final.md` match `results/thesis_results.md` and `results/tables/`.
3. **Pass C – Interpretive consistency**: verify that narrative text is consistent with the sign, direction, and significance of results.
4. **Pass D – Pipeline provenance**: confirm that all result files were generated by the same pipeline run.

Do not skip passes. Each pass produces a checklist output. Violations must be resolved before committing thesis text.

## PART 2 — HARD RULES

These constraints must never be violated.

**CR-1: Single pipeline run.** All result files cited in a single thesis version must originate from the same pipeline run. The `results/pipeline_run_log.jsonl` file records run timestamps and file hashes. Mixed-run results are a protocol violation.

**CR-2: Numeric precision.** All numbers reported in `thesis_final.md` must be rounded to four decimal places and must exactly match the corresponding value in the authoritative source file at that precision. Rounding artefacts introduced by manual transcription are a violation.

**CR-3: Significance threshold consistency.** The declared α level (0.05) must be applied uniformly. A result described as "significant" must satisfy p < 0.05; a result described as "not supported" must have p ≥ 0.05. Mixed verdicts within the same specification block are a violation.

**CR-4: Direction consistency.** Every narrative statement about the direction of an effect (risk-reducing, risk-increasing, positive, negative) must be consistent with the sign of the reported coefficient. A negative β accompanied by the phrase "associated with higher NRS" is a violation.

**CR-5: Verdict consistency.** H1 and H2 verdicts stated in Sections 5.5, 5.7, 5.8, 6.2, and 6.3 must all agree. Divergent verdicts across sections are a violation.

**CR-6: Stale placeholder detection.** Any `<!-- RESULTS:BEGIN:… -->` block in `thesis_final.md` must contain content that matches the corresponding block in `results/thesis_results.md`. Blocks that differ indicate the thesis has not been updated after the most recent pipeline run.

**CR-7: Alignment rate interpretation.** An alignment rate below 0.50 must never be described as indicating directional consistency with sentiment. The threshold is 0.50; rates below it indicate contrarian or non-aligned response patterns.

**CR-8: Composite vs. component attribution.** Narrative claims about "the Shock Score effect" must reference SC_total coefficients, not individual component coefficients. Component coefficients (ac_e, se_e, ai_e, es_raw) are robustness results; they may support or qualify the composite finding but must not be used as the primary evidence for H1.

## PART 5 — PASS C: INTERPRETIVE CONSISTENCY

Interpretive consistency checks cannot be automated. They require reading the relevant narrative passages and applying the rules below.

### C1. Direction narrative check

For every coefficient discussed in Sections 5.5 and 5.6, confirm:
- Negative coefficient → language uses "risk-reducing", "lower NRS", "decreasing risk exposure", or equivalent
- Positive coefficient → language uses "risk-increasing", "higher NRS", "increasing risk exposure", or equivalent
- A coefficient described as "consistent with loss aversion" must be **negative** (loss aversion predicts risk reduction under threat)
- Exception: the Spec 5 interaction term (`SC_total × D_neg`) is positive and represents **amplification** of loss aversion for negative events; this must be described as amplification, not as a contradiction

### C2. Significance language check

| p-value range | Required language | Prohibited language |
|--------------|-------------------|---------------------|
| p < 0.001 | "statistically significant", "p < 0.0001" | "marginally significant", "trending" |
| 0.001 ≤ p < 0.05 | "statistically significant at α = 0.05" | "highly significant" |
| 0.05 ≤ p < 0.10 | "not significant at α = 0.05"; may note marginal tendency descriptively | "significant", "supported" |
| p ≥ 0.10 | "not statistically significant" | "approaching significance" |

### C3. H2 split-outcome check

When H2 is not supported on the primary outcome (portfolio_return) but significant on a secondary outcome (sharpe_ratio), both facts must appear in Section 5.7. The primary verdict is "not supported". The secondary finding is reported as directionally supportive. Neither outcome should be suppressed.

### C4. ES_raw sign interpretation check

The positive ES_raw coefficient in Spec 3 must be present in the thesis with an interpretation. Acceptable interpretations include:
- Contrarian-resolution mechanism (category-level expectation adjustment)
- Value-at-risk recalibration under known-volatility event types

It must **not** be described as inconsistent with H1, as contradicting SC_total, or as a data error.

### C5. Composite–component coherence check

The overall H1 narrative must rely on the SC_total primary coefficient. Component results (Spec 3) are supplementary. Sentences of the form "H1 is supported because AI_e is negative" are a violation; the correct form is "H1 is supported because SC_total is significant; the component decomposition further shows…".

## PART 6 — PASS D: PIPELINE PROVENANCE

### D1. Hash verification

`results/pipeline_run_log.jsonl` records SHA-256 hashes of key output files at the time of each run. After any pipeline run, verify:

```
pca_diagnostics.json  hash matches log entry
tbl_h1_main.csv       hash matches log entry
tbl_h2_portfolio.csv  hash matches log entry
thesis_results.md     hash matches log entry
```

If hashes do not match, one or more files were modified after the pipeline run. Identify the modification and either re-run the pipeline or document the manual change with a GitHub issue.

### D2. Scenario shock score version check

`scenario_shock_score.csv` is the upstream input to both `3_survey_assembly.py` and `8_statistical_analysis.py`. Confirm the file has not been modified between the assembly run and the analysis run by comparing the `mtime` of `scenario_shock_score.csv` against the timestamps in the pipeline log. If the shock score CSV is newer than the analysis output, re-run `8_statistical_analysis.py`.

### D3. Project knowledge cache notice

The Claude project knowledge cache may reflect a prior state of `diagnostics/survey_assembly_report.md` and `results/thesis_results.md`. When a discrepancy is observed between project-knowledge values and live repository values, the **live repository is authoritative**. Project knowledge cache discrepancies are not formula errors and do not require code changes.
